## 一、面试心态建立

### 1.1 面试官不是你敌人

面试官的核心需求是：**找到一个能干活、不出事、能沟通的同事**。

他不是来挑你毛病的。他是来找"能用的人"的。你只需要证明三点：
1. **我会这个**（技术过关）
2. **我知道为什么这么做**（有思考）
3. **我不会搞出生产事故**（有工程素养）

### 1.2 遇到不会的问题怎么办

- ❌ 死记硬背答案 → 被追问两次就露馅
- ❌ 直接说"不知道" → 放弃展示机会
- ✅ **诚实承认 + 展示你的思考框架**： "这个我确实没深入了解过，但如果让我来设计，我会先考虑 X 问题，再考虑 Y 问题，不知道这个思路对吗？"

### 1.3 五个"一定避免"的回答

| 不要说 | 为什么 | 改成 |
|--------|--------|------|
| "因为大家都这么用" | 你没思考 | "因为在这个场景下，X 比 Y 有这些优势..." |
| "我本地能跑啊" | 你不懂环境差异 | "我们在 Docker 中固化所有依赖，保证任何环境一致" |
| "这个很简单" | 你轻浮 | "这个问题的核心在于..." |
| "应该是" | 你不确定还不查 | "我记得是 X，但我需要确认一下" |
| "就是文档上写的" | 你没有深入理解 | 讲你自己的理解 |

## 三、面试官的三个致命追问（和你的必杀回答）

### 追问 1："你这个系统真的能跑吗？"

必杀回答："每个项目都有独立的 `docker compose up` 一键启动命令 + `run_tests.sh` 一键测试脚本。验证者不需要修改任何源码，不需要了解任何内部实现，只需要两行命令就能验证全部功能。"

### 追问 2："你选的技术栈不是最好的"

必杀回答："技术选型从来没有'最好'，只有'最合适这个场景'。我选 Rust 不是因为 Rust 比 Python 好，而是因为 Token 计数是 CPU 密集型计算，Rust 比 Python 快 10 倍，这个差距在 Agent 场景下是用户体验关键。如果我做的是一个管理后台的增删改查，我也会选 Python。"

### 追问 3："你一个人怎么做了 7 个项目？"

必杀回答："7 个项目不是 7 个独立的复杂系统。token-core 是纯计算引擎（无依赖），mcp-bridge 是协议转换层（无状态），AgentViz 是前端组件库（npm 包）。它们各自职责单一、边界清晰。真正复杂的只有 AgentForge 和 AgentEval-Platform 两个项目。而且每个项目都有充分的自动化测试保障。"
