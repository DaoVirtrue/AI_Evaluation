## 一、为什么做这个项目（业务需求）

### 1.1 背景

2024-2026 年，AI Agent 从"概念验证"走向"生产落地"。企业面临三个核心痛点：

**痛点 1：Agent 是黑盒，无法评测**
- Agent 调用链长达 5-20 步（LLM → Tool Call → LLM → ...），出错了不知道哪一步的问题
- 算法团队改了一行 System Prompt，不知道是变好了还是变坏了
- 需求：一个能创建评测、自动执行、对比版本、回放轨迹的平台

**痛点 2：Token 成本失控**
- 2026 年主流模型价格差 500 倍（DeepSeek V4 Flash $0.14/百万 vs Fable 5 $50/百万）
- 缓存折扣（省 90%）、长上下文溢价（翻 2 倍）、思考 Token 独立计费——没人能手动跟踪 16 个模型的定价规则
- 需求：一个自动精确计量 Token 和成本的引擎

**痛点 3：工具调用碎片化**
- OpenAI 用 Function Calling，Anthropic 用 Tool Use / MCP，每个工具只为一种协议编写
- 换模型 = 工具全部重写
- 需求：一个协议桥接层，让任何 Agent 调用任何工具

### 1.2 目标用户

- **算法工程师**：需要快速评测 Agent 版本、对比效果、定位问题
- **平台工程师**：需要统一 Agent 接口、管理工具调用、控制成本
- **团队 Tech Lead**：需要审计 Token 消耗、优化模型选择、降低 API 成本

## 三、精准度问题及解决路径（核心面试亮点）

### 3.1 问题 1：Token 计数的编译时精度

**问题**：Rust PyO3 绑定编译失败，Docker 构建 10+ 次才成功。根本原因是 Rust 1.82 的编译器无法处理新 crate 版本。

**精确错误链**：
1. `toml_datetime v1.1.1+spec-1.1.0` 解析失败 → Rust 1.82 太旧
2. 升级到 `rust:latest-slim` 后：`{:.4f}` 格式非法 → Rust 用 `{:.4}` 不是 C printf 风格
3. `f64.max(1)` 类型不匹配 → 改为 `(as usize).max(1)`
4. `Vec<String>` 不能从 `Vec<&str>` 构建 → `p.id.to_string()`
5. `From<TokenError> for PyErr` 孤儿规则 → 用 `tc_try!` 宏替代
6. `WindowStatus` 缺少 `Serialize` derive → 添加后重复 derive 冲突
7. `wasm-bindings` workspace 成员不存在 → `sed` 移除

**解决路径**：逐错误修复 → 最终 Rust 编译成功 → `import token_core_py` 返回 17 个模型。

**面试回答**：这就是 Rust 的 trade-off——编译时严格保证运行时安全，但编译过程确实需要更多调试。每个错误都是 Rust 编译器在帮你防止潜在的运行时 bug。

### 3.2 问题 2：模型成本对比排序错误

**问题**：Rust `compare_models` 未按价格升序排列，Playwright 测试失败。

**原因**：Rust 版使用 `filter_map` 直接收集结果，保持了输入顺序而非成本顺序。

**解决**：加一行 `results.sort_by(|a, b| a.estimated_cost.partial_cmp(&b.estimated_cost).unwrap_or(Equal))`。

**面试回答**：这是"细节决定准确性的典型案例。一个排序遗漏导致 Playwright 测试断言失败。Rust 比 Python 更需要显式处理浮点比较（`partial_cmp`），但这恰恰保证了不会出现 NaN 导致的静默错误。"

### 3.3 问题 3：评测 Worker 的并发精度

**问题**：FastAPI 的 asyncio 事件循环与 Worker 线程中的 `asyncio.new_event_loop()` 冲突，导致 `RuntimeError: Task got Future attached to a different loop`。

**原因**：Python 的 asyncio 不是线程安全的，不同线程不能共享事件循环。

**解决**：Worker 改为同步模式——用 `requests` + `psycopg2` 同步引擎，独立线程运行，不和 FastAPI 的 async 引擎交互。

**面试回答**：这是 Python 异步编程的经典陷阱。解决方案是"关注点分离"——FastAPI 用 async 处理 HTTP 请求（I/O 密集），Worker 用 sync 处理评测执行（CPU 密集 + 外部调用），两个引擎独立运行互不干扰。

### 3.4 问题 4：Mock 评测的准确性问题

**问题**：没有 API Key 时，Agent 返回 `"[Native/model] Processed N messages"` 这种无意义回答，评测通过率只有 20%。

**原因**：适配器是占位实现，不包含任何知识。

**解决**：在 AgentForge 的 NativeAdapter 中实现智能 Mock——内置 20+ 条知识库覆盖常见问题（地理、科学、编程、DevOps），支持简单算术表达式求值。Mock 模式自动检测是否有 API Key，有 Key 时切换真实 LLM。

**结果**：通过率从 20% → 100%。

**面试回答**：这是一个"可用性优先"的工程决策。Demo 需要在没有外部依赖的情况下完整演示评测闭环。智能 Mock 不是"假的"，而是"降级方案"——和真实 LLM 的接口兼容，配 Key 后无缝切换。

### 3.5 问题 5：数据集的持久化精度

**问题**：上传的数据集只存了 `preview_data`（前 5 条），完整文件在容器重启后丢失。

**解决**：
1. Docker volume 挂载 `./datasets:/app/datasets`
2. API 上传时写文件到磁盘
3. Worker 从磁盘读取完整数据集
4. 启动时自动导入 `sample_qa.jsonl`

**面试回答**：这体现了"数据完整性"的工程意识。元数据存数据库（高效查询），文件存磁盘（完整保留），两者互补。Docker volume 保证了容器重启不丢数据。

### 3.6 问题 6：前端业务组件的完整性

**问题**：初版前端只有 3 个字段（名称/模型/数量），Dashboard 只显示名称和状态，缺少数据集选择、结果表格、统计卡片、筛选功能。

**解决**：5 个完整页面重写：
1. Dashboard — 4 统计卡片 + 表格（名称/模型/状态/通过率/进度/时间/操作）
2. 创建评测 — 数据集选择+预览 + Agent 配置 + 评分方式
3. 评测详情 — 6 指标卡片 + 结果表格 + 全部/通过/失败筛选
4. 数据集 — 上传 + 列表 + 预览弹窗 + 删除
5. 后端 — Dataset CRUD + 评测 Results/Stats API

**面试回答**：这是"从 Demo 到产品"的跨越。初版只验证了技术可行性，补全后才有业务闭环——从准备数据到创建评测到查看分析，用户能独立完成整个工作流。

## 四、面试高频追问的回答框架

### Q: 你遇到的最大技术挑战是什么？

A: "Rust PyO3 的 Docker 编译。Rust 的严格编译在保证运行时安全的同时，让构建过程需要解 7 个错误。最终在 Docker 内成功编译，`backend: rust` 运行在生产中。这体现了'为正确性付出编译时成本'的工程哲学。"

### Q: 没有 API Key 怎么验证系统？

A: "智能 Mock 模式。AgentForge 内置知识库，关键词匹配回答常见问题。同时接口和真实 LLM 完全兼容——配 Key 后自动切换。Mock 不是为了造假，是为了在没有外部依赖时也能验证系统完整性。"

### Q: 精度问题你怎么保证的？

A: "多层验证：Rust 编译时类型检查 → Pydantic 运行时校验 → Playwright E2E 浏览器验证 → 成本计算用 Decimal 避免浮点误差。每一层解决不同维度的精度问题。"
