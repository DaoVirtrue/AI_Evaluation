## 第一部分：是什么

### 1.1 生活类比：行车记录仪 + 导航地图

你开车从北京到上海，如果只有一个"到了"的通知，你永远不知道为什么开了 12 小时（堵车？绕路了？走错了？）。

你需要**行车记录仪**——每一步都有记录，可以回放看"在哪个路口做错了决定"。

AgentViz 就是 Agent 的"行车记录仪 + 导航地图"：
- **轨迹时间线** = 行车记录仪，回放 Agent 的每一步推理
- **Token 消耗图** = 油耗表，每一步花了多少"钱"
- **工具调用图** = 导航路线，Agent 调用了哪些工具、哪个环节卡住了

### 1.2 大白话解释

**什么是"轨迹"（Trajectory）？**

Agent 解决问题不是一步到位的。比如你问"帮我搜一下今天北京的天气，然后发邮件给老板"：

```
步骤1: LLM 调用 → 理解你的意图
步骤2: 工具调用 → 调用 web_search("北京天气")
步骤3: LLM 调用 → 分析搜索结果
步骤4: 工具调用 → 调用 send_email("老板", content)
步骤5: LLM 调用 → 确认发送成功
步骤6: 输出 → "已完成！"
```

这 6 步就是一条"轨迹"。如果不可视化，你只能看到最后的"已完成！"，看不到中间哪一步慢了、哪一步错了。

**什么是 Canvas 和 DOM？**

| 概念 | 类比 | 适合场景 |
|------|------|---------|
| DOM | 乐高积木 | 几十到几百个步骤，每个都可以点击 |
| Canvas | 画布颜料 | 几千到几万个步骤，只看不点 |

**为什么需要两种渲染方式？**

DOM 方式：10 个步骤 → 每个步骤生成一个 `<div>` 元素 → 很快
DOM 方式：10000 个步骤 → 生成 10000 个 `<div>` → 浏览器卡死

Canvas 方式：直接往画布上"画"内容，不生成 DOM 节点 → 10000 个步骤也能 60 帧/秒

**AgentViz 自动判断**：少于 1000 步用 DOM（方便交互），超过 1000 步自动切换 Canvas（保证流畅）。

### 1.3 8 个可视化组件的"一句话"

| 组件 | 一句话 | 解决什么问题 |
|------|--------|------------|
| TrajectoryTimeline | Agent 推理步骤时间线 | "Agent 在哪一步做错了？" |
| ToolCallGraph | 工具调用依赖关系图 | "哪个工具调用失败了？重试了几次？" |
| ContextWindowViewer | 上下文窗口填充状态 | "AI 的记忆还剩多少空间？" |
| TokenUsageChart | Token 消耗趋势 | "哪一步花钱最多？" |
| AgentDiffView | 两次运行差异对比 | "新版本为什么比旧版本差？" |
| ThinkingStreamPlayer | 思维流逐字回放 | "AI 在想什么？" |
| LatencyWaterfall | 延迟瀑布图 | "哪一步最慢？" |
| ErrorTraceView | 错误追踪 | "出错了，当时 AI 看到了什么？" |

## 第二部分：为什么

### 2.1 为什么不直接用后端返回的 JSON 数据？

后端返回的 Agent 轨迹是这样的：

```json
{"steps":[{"type":"llm_call","tokens":500}, {"type":"tool_call","tool":"search"}]}
```

人能看懂吗？能，但很痛苦。面试官要的不是"你能读 JSON"，而是"你能把数据变成洞察"。

可视化的价值在于**降低认知负担**：
- 一眼看出哪步最慢 → 不需要逐行对比数字
- 一眼看出 Token 在哪步炸了 → 不需要心算加法
- 一眼看出两个版本的区别 → 不需要人工 diff JSON

### 2.2 为什么用 TypeScript 而不是 JavaScript？

| 维度 | JavaScript | TypeScript |
|------|-----------|-----------|
| 类型检查 | 无（运行时才报错） | 编译时就报错 |
| 重构安全性 | 改了组件名 → 不知道哪里坏了 | 改了组件名 → 编译器告诉你要改哪里 |
| 团队协作 | 看代码靠猜 | 类型即文档 |

AgentViz 是一个组件库（npm 包），会被其他开发者使用。如果用 JavaScript，别人不知道你的组件接受什么参数。TypeScript 的类型定义就是"使用说明书"。

**为什么不用 React 之外的选择（Vue/Svelte/Angular）？**

React 生态最成熟、npm 下载量最高、第三方组件库最丰富。作为要给其他项目使用的通用组件库，选最主流的技术栈可以最大化复用范围。

### 2.3 为什么需要 Web Worker 处理数据？

浏览器渲染页面用的是"主线程"。如果主线程被数据聚合计算占用了 500 毫秒，这 500 毫秒内用户点击任何按钮都没反应（页面"卡死"）。

Web Worker 是"后台线程"——在另一个线程里做数据计算，主线程只负责渲染。这样用户操作永远流畅。

**类比**：餐厅里厨师（主线程）专心做菜（渲染页面），帮厨（Web Worker）在后台洗菜切菜（数据预处理）。厨师不用等帮厨，帮厨干完了直接把备好的菜递给厨师。

### 2.4 如何实现硬性要求

**低延迟**：这是前端项目最核心的指标。Canvas 渲染 10w+ 步骤 → 60fps；react-window 虚拟滚动 → 只渲染可见区域；Web Worker 预处理 → 主线程零阻塞。

**高可用（前端版）**：ErrorBoundary 错误边界——组件内部崩溃不影响整个页面，显示一个优雅的替换 UI 而不是白屏。Canvas 不支持时自动降级为 DOM 渲染。

**可扩展**：每个可视化组件是独立的 npm 包（`@agent-viz/core`、`@agent-viz/react`），按需导入。新增组件不影响已有组件。

**安全**：不使用 `eval()`（动态执行代码的函数）、不使用 `dangerouslySetInnerHTML`（React 中直接插入 HTML 的方式）。所有用户输入的轨迹文本用 DOMPurify 清洗后渲染——XSS 攻击无效。

## 第三部分：怎么做

### 3.1 白话实现路径

```
1. 数据到达 → @agent-viz/transform 在 Web Worker 中标准化轨迹格式
              （不管来自 AgentForge 还是 LangChain，格式统一）
         ↓
2. 判断数据量 → <1000 条用 DOM 渲染（React 组件） 
              → >1000 条用 Canvas 渲染（直接在画布上画）
         ↓
3. 虚拟滚动 → react-window 只渲染可视区域内的步骤
              （屏幕外的不生成 DOM 节点，节省内存）
         ↓
4. 用户交互 → 点击某个步骤 → 展开详情（输入/输出/Token/延迟）
            → 拖拽缩放时间线
            → 播放思维流
```

### 3.2 关键实现的"为什么"

**为什么要用 monorepo（一个仓库放多个 npm 包）？**

四个包（core、react、vanilla、transform）共享类型定义，如果分在四个仓库，改一个类型要提交四次、发四次包。monorepo 只需要一次修改。

**为什么要用 Storybook？**

Storybook 是一个"组件目录"——把每个组件独立展示出来，可以在不集成到 AgentEval-Platform 的情况下单独开发和调试。类似于"汽车零部件测试台"——不需要装到整车上就能单独测试发动机。

## 第四部分：面试指南

### Q1: Canvas 和 DOM 有什么区别？什么时候用哪个？

DOM 是浏览器内置的元素树，每个元素都是独立的对象，可以绑定事件、修改样式，但多了会卡。Canvas 是一块像素画布，你在上面"画画"，没有独立的对象，性能好但不方便交互。AgentViz 用"自适应渲染"——数据量少用 DOM（方便点开看详情），数据量大用 Canvas（保证不卡）。

### Q2: 虚拟滚动是什么？为什么需要？

当你有一万条数据要显示在一个 600px 高的区域里，如果每条生成一个 30px 的 DOM 节点，总高度 30 万像素。浏览器需要处理所有这些节点，即使大部分在屏幕外面看不见。虚拟滚动就是"只渲染看得见的那部分"——屏幕外的数据不生成 DOM 节点，只保留一个占位的高度。这样 1 万条数据和 20 条数据的性能几乎一样。

### Q3: 如何防止前端 XSS 攻击？

前端 XSS 的核心是"用户输入被当成代码执行"。AgentViz 的轨迹文本可能来自任意 Agent 的输出——如果 Agent 输出了 `<script>alert('hack')</script>`，我们不能直接插入 DOM。解决方案：所有文本显示前经 DOMPurify 清洗，移除所有可执行代码。不使用 `dangerouslySetInnerHTML`。不使用 `eval()`。

### Q4: 你这个组件库和 ECharts/D3.js 的关系是什么？

ECharts 和 D3.js 是底层图表库，提供"画柱状图""画折线图"等通用能力。AgentViz 是上层业务组件，专门解决"Agent 轨迹"这个特定领域的可视化需求。AgentViz 内部可能用 D3.js 做图布局（如工具调用 DAG），用 Canvas API 做时间线渲染，但它封装了"Agent 领域知识"——比如 Context Window Viewer 知道 System Prompt 要用紫色、Tool Result 要用黄色，这些是通用图表库不会做的。

### Q5: 怎么测试可视化组件？看起来没法做单元测试？

可视化组件测试分两层：一是用 Vitest + jsdom 做渲染测试——验证组件在给定数据下渲染出正确的 DOM 结构（比如传入 5 个步骤，应该渲染出 5 个时间线节点），不验证"好不好看"。二是用 Storybook + Chromatic 做视觉回归测试——截图对比，检测渲染是否有意外的变化。好看由设计师验证，行为正确由测试验证。
