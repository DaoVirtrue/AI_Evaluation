# 01 — 开篇：7 个项目全景图 + 面试心态

> **阅读对象**：零基础小白。读完这篇，你将理解这 7 个项目为什么存在、它们之间是什么关系、面试官真正想考察什么。

---

## 一、先讲一个故事

假设你是美团的外卖配送系统负责人。

一开始，你只有一个骑手（一个 Agent），只要告诉他"去 A 餐厅取餐，送到 B 小区"就行。很好管。

但后来你有了 100 个骑手，每个骑手能力不同（有人骑电动车快、有人骑自行车慢），餐厅分布在不同的区域，有的订单需要同时取多个餐厅。这时候问题来了：

1. **你怎么知道哪个骑手最适合这笔订单**？（类比：ModelRouter 智能模型路由）
2. **你怎么知道骑手有没有偷懒、走错路**？（类比：Agent 评测平台）
3. **骑手说"路上堵车"是真的还是借口**？（类比：轨迹回放调试）
4. **你怎么统计今天花了多少配送费**？（类比：Token 成本计算）
5. **如果配送站服务器挂了，怎么保证订单不丢**？（类比：高可用部署）

这 7 个项目，就是分别解决这 5 类问题的。

---

## 二、7 个项目像一家公司的 7 个部门

用"开一家公司"来类比这 7 个项目：

| 项目 | 公司部门类比 | 干什么的 |
|------|------------|---------|
| **token-core** | 财务部 | 计算每一笔开销（Token 消耗 = 钱），确保不超预算 |
| **mcp-bridge** | 外联部 | 和外部供应商（工具）打交道，统一沟通协议 |
| **AgentViz** | 数据可视化部 | 把晦涩的数据画成图表，让人一眼看懂 |
| **AgentForge** | 生产部（核心） | 接订单、安排工人（Agent）、保证质量 |
| **AgentEval-Platform** | 质检部 | 检测每个工人干得怎么样，好的留下、差的改进 |
| **agent-container-service** | 后勤/IT部 | 保证办公室（服务器）正常运转、水电不断 |
| **claude-code-toolkit** | 效率工具组 | 给员工配好用的工具，让干活更快 |

**一句话串起来**：AgentForge 生产 Agent → token-core 算成本 → mcp-bridge 调用工具 → AgentEval-Platform 检测质量 → AgentViz 把结果画出来 → agent-container-service 保证服务不挂 → claude-code-toolkit 让开发者写代码更快。

## 三、学习的正确顺序（为什么不是 1→2→3→...→7）

零基础学习编程有个大坑：**按编号顺序学**。比如先学"前后端分离"再学"什么叫 API"——这就好比先学开车再学方向盘是什么。

正确顺序是**从底层到上层，从简单到复杂**：

```
第1步：token-core     ← 最简单，纯计算，没有依赖，先理解"Token是什么"
第2步：mcp-bridge      ← 理解"协议"——计算机之间怎么说话
第3步：AgentViz        ← 理解"前端"——怎么把数据变成人看得懂的图
第4步：AgentForge      ← 理解"框架"——把前面三个串起来
第5步：AgentEval-Platform ← 理解"全栈"——前后端协作
第6步：agent-container-service ← 理解"部署"——代码怎么跑在服务器上
第7步：claude-code-toolkit ← 理解"开发工具"——怎么让自己写代码更快
```

**类比**：学盖房子——
- 先认识砖头水泥（token-core = 基础材料）
- 再学水管电线怎么接（mcp-bridge = 标准化接口）
- 再学装修设计（AgentViz = 可视化）
- 再学怎么把砖头水泥水管电线组合起来（AgentForge = 框架）
- 最后学怎么验收房子、物业管理（AgentEval-Platform = 平台）

## 四、面试官到底想考察什么

**不是你背了多少概念，而是你理解背后的"为什么"**。

面试官的三个核心问题（每个项目都要过这三关）：

| 面试官问 | 他真正想知道什么 | 你的回答结构 |
|---------|----------------|------------|
| "你为什么用 Rust 而不是 Python？" | 你有技术选型思维，不是跟风 | 场景对比 + 量化理由 |
| "你这个系统怎么保证高并发？" | 你理解瓶颈在哪里，怎么解决 | 指出瓶颈 + 方案 + 代价 |
| "如果服务挂了怎么办？" | 你有容灾意识，不是只写 happy path | 预防 + 检测 + 恢复 |

## 六、这篇文档的总结

在继续阅读后续 8 篇之前，记住这三句话：

1. **每个技术都是被逼出来的**——没有"高并发需求"，就不会有人发明"连接池"
2. **技术选型没有银弹**——每个选择都有代价，你需要知道代价是什么
3. **面试不是考试，是对话**——你不需要背答案，你需要理解背后的逻辑

---

## 面试话术速记

- 问："介绍一下你的项目" → 答："7 个项目构成一个 Agent 全生命周期平台，从 Token 计算到 Agent 执行到评测到部署。核心是 AgentForge，token-core 是它的计量引擎，mcp-bridge 是它的工具网关，AgentEval 是它的质检平台。"
- 问："你为什么这么设计" → 答："按照单一职责原则，每个项目只做一件事。token-core 只算 Token，不调 API；AgentForge 只调度 Agent，不存评测数据。好处是每个项目可以独立升级、独立测试、独立部署。"
