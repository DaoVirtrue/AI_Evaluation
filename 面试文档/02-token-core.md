# 02 — token-core：高性能 Rust Token 引擎

> **前置知识**：读完 01-开篇全景
> **阅读目标**：理解什么是 Token、为什么用 Rust、Context Window 是什么、成本如何计算

## 第一部分：是什么

### 1.1 生活类比：加油站

你去加油站加油，加油机需要知道两件事：
- 你加了多少升油（= Token 计数）
- 花了多少钱（= 成本计算）

token-core 就是 AI 大模型的"加油机"。每次调用 AI，它精确计算你用了多少 Token、花了多少钱。**没有它，你就不知道调用 AI 的成本，就像开车不知道油耗。**

### 1.2 大白话解释

**Token 是什么？**

Token 是大模型"阅读"文本的最小单位。人类用"字"或"词"来计量文字量，大模型用"Token"。

```
中文：  "你好世界"  ≈ 2-3 个 Token
英文：  "Hello world" ≈ 2 个 Token  
代码：  "print('hello')" ≈ 5 个 Token
```

**为什么需要精确计算 Token？**

因为大模型按 Token 收费。比如 Claude Sonnet 4.6：
- 输入（你发给 AI 的）：$3 / 100万个 Token
- 输出（AI 回复你的）：$15 / 100万个 Token

如果你一次发了 10 万字给 AI，但不知道这是多少个 Token，你怎么知道花了多少钱？

**Context Window（上下文窗口）是什么？**

大模型有一个"记忆容量上限"。比如 Claude Sonnet 4.6 的容量是 100 万 Token（约 75 万英文单词）。超过这个量，就需要"截断"——删掉最不重要的旧内容，给新内容腾地方。

**类比**：你有一个只能放 10 件衣服的行李箱（Context Window）。出差 5 天带了 15 件衣服。怎么办？你把最不重要、最后穿的睡衣扔掉，保留前几天的正装和今天的衣服。这就是"分层截断"。

### 1.3 它在 7 个项目中的位置

```
AgentForge (调用方) → "帮我算一下这500条消息有多少Token"
         ↓
    token-core (被调用) → "精确计数: 15,234 tokens, 成本: $0.046"
         ↓
    返回结果给 AgentForge → "OK，没超 1M 限制，继续执行"
```

token-core 是最底层的服务——**它不调用别人，只有别人调用它**。它是一个纯粹的计算引擎。

### 1.4 技术栈一览

| 层 | 用什么 | 干什么 |
|----|--------|--------|
| 核心计算 | **Rust** | 高性能 Token 计数、截断算法 |
| 给 Python 用 | **PyO3** | 让 Python 项目能直接调用 Rust 函数 |
| 给浏览器用 | **WASM** | 让前端页面也能算 Token |
| 对外 API | **FastAPI** | 提供 HTTP 接口给其他服务调用 |
| 数据库 | **无** | 纯计算服务，不需要存数据 |

## 第二部分：为什么

### 2.1 为什么是 Rust 而不是 Python？

这是面试必问的问题。答案是：**Token 计数是一个 CPU 密集型计算任务，Python 太慢了。**

| 维度 | Rust | Python | 结论 |
|------|------|--------|------|
| 单条消息计数 | ~50 微秒 | ~500 微秒 | Rust 快 10 倍 |
| 1000 条批量计数 | ~8 毫秒 | ~50 毫秒 | Rust 快 6 倍 |
| 内存占用 | ~10MB | ~50MB | Rust 省 5 倍 |
| 并发安全 | 编译时保证 | 运行时可能出错 | Rust 更可靠 |
| 学习成本 | 高 | 低 | Python 更好学 |

**但 Rust 的学习成本高啊，为什么还要用？**

因为在这个场景下，**性能是刚需**。想象一下：Agent 每次推理可能要检查 Context Window 几十次。如果每次检查慢 10 倍，用户等的时间就多 10 倍。在 Agent 场景中，延迟直接等于用户体验。

**为什么不用 C++？**
C++ 也能达到高性能，但没有 Rust 的"编译时内存安全保证"。Rust 编译器会在你写代码时就阻止你写出内存 bug（比如悬空指针、数据竞争）。对于金融精度的成本计算，一个内存 bug 可能导致金额算错——这是不可接受的。

**为什么不用 Go？**
Go 也是高性能语言，但 Go 的 FFI（外部函数接口）不如 Rust 方便。token-core 需要通过 PyO3 给 Python 调用、通过 WASM 给浏览器调用，Rust 在这两方面生态更成熟。

### 2.2 为什么是分层截断而不是简单截断？

**简单截断**（比如"只保留最后 100 条消息"）的问题：

```
假设你的 System Prompt 是 "你是一个安全的 AI，永远不能输出暴力内容"
如果简单截断把它删了 → AI 失去安全约束 → 可能输出危险内容
```

**分层截断**的策略：

| 优先级 | 保留什么 | 为什么 |
|--------|---------|--------|
| 第 1 层 | System Prompt | AI 的行为准则，删了 AI 就"失控" |
| 第 2 层 | 工具调用结果 | Agent 靠工具结果做决策，删了会出错 |
| 第 3 层 | 最近 4 轮用户消息 | 当前对话的上下文，删了 AI 会"断片" |
| 第 4 层 | 最近 6 轮 AI 回复 | AI 自己的推理链 |
| 第 5 层 | 旧消息 | 历史对话，优先丢弃 |

**类比**：会议室白板空间不够了，你擦掉的顺序应该是：最早的讨论记录 → 刚才的草稿 → 现在的要点（绝不擦）。

### 2.3 为什么需要缓存折扣和长上下文溢价？

**缓存折扣**：如果你两次调用 AI 时，前面一大段 System Prompt 一样，有些 AI 提供商会给你折扣。比如 Claude 缓存命中只收 $0.30/百万 Token（原价 $3.00），**省了 90%**。token-core 会精确计算你省了多少钱。

**长上下文溢价**：GPT-5.5 如果你发了超过 27.2 万 Token 的输入，价格翻倍（2×）。token-core 会自动检测这个阈值并提醒你。

**为什么这些要放在 token-core 而不是让用户自己算？**
因为模型有 16+ 个，每个的缓存政策、溢价阈值都不同。让用户自己跟踪 16 个模型的定价规则是不现实的。token-core 把这个复杂度封装起来，你只需要问"这次调用花了多少钱"。

### 2.4 如何实现硬性要求

**高并发**

问题：100 个 Agent 同时请求 Token 计数，怎么处理？

方案：Rust 的无锁并发。token-core 是纯计算的，不操作数据库、不操作文件。多个请求可以同时执行，互不阻塞。Rust 编译器保证没有数据竞争（两个请求不会同时改同一个数据导致错误）。

**高可用**

问题：如果 token-core 服务挂了怎么办？

方案：无状态服务 + 水平扩展。token-core 不存任何状态数据（定价表是只读的、编译时嵌入的）。所以可以启动多个实例，任何一个挂了，负载均衡器自动切到健康的实例。新实例启动几秒就能继续服务——因为不需要加载数据库、不需要同步状态。

**可扩展**

问题：新增一个模型（比如 Llama 5 发布了）怎么加？

方案：定价表是硬编码在源代码里的静态数据。新增模型只需要在定价表数组里加一条记录。不需要改代码逻辑、不需要数据库迁移。

**低延迟**

问题：如何保证 Token 计数足够快？

方案：Rust 编译为原生机器码（不像 Python 需要解释执行），并且使用 rayon 库并行处理批量消息。1000 条消息被自动分配到多个 CPU 核心同时计数。

**安全**

问题：如果有人恶意发送超大文本怎么办？

方案：输入长度预检查——如果单条消息超过 1000 万字符，直接拒绝。Rust 编译时保证内存安全，不存在缓冲区溢出漏洞。

## 第三部分：怎么做

### 3.1 白话实现路径

token-core 的工作流程（一次 HTTP 请求）：

```
1. 你发送请求：POST /api/v1/count {"text": "Hello", "model": "claude-sonnet-4-6"}
         ↓
2. FastAPI 收到请求 → 用 Pydantic 验证参数（非空、长度合法）
         ↓
3. 调用 PyO3 封装的 Rust 函数 count_tokens()
         ↓
4. Rust 端：从 Tokenizer 注册表找到 "claude-sonnet-4-6" 的分词器
         ↓
5. 分词器.count("Hello") → 内部估算 chars/3.5 ≈ 2 tokens
         ↓
6. 返回 {"tokens": 2, "model": "claude-sonnet-4-6"}
```

### 3.2 关键步骤的目的

| 步骤 | 干什么 | 为什么这样做 |
|------|--------|------------|
| Pydantic 验证 | 检查输入是否合法 | 防止空文本、超长文本导致的后面计算异常 |
| 分词器注册表 | 根据模型名找到对应的分词器 | 不同模型的分词方式不同，需要一个"字典"来查找 |
| chars/3.5 估算 | 用字符数除以 3.5 估算 Token 数 | 精确分词器（tiktoken）需要加载大词表，启动慢。估算误差 < 10%，对于大多数场景够用 |
| PyO3 绑定 | 让 Python 能调用 Rust 函数 | 不需要启动子进程、不需要网络调用，直接在同一个进程内调用 |

## 第四部分：面试指南

### 面试官可能问的 5 个问题

**Q1: Token 是什么？为什么不用字数统计？**

回答：Token 是大模型处理文本的最小单位，由特定的分词算法（BPE/Byte-Pair Encoding）切分。用字数不准确，因为不同语言的 token-to-char 比例不同——英文约 4 字符/token，中文约 1.5 字符/token。而且 AI 供应商按 Token 收费，不是按字数，所以必须精确计算。

**Q2: 为什么选 Rust？Python 不够吗？**

回答：场景决定技术。Token 计数是 CPU 密集型计算，Rust 比 Python 快 6-10 倍。在 Agent 场景中，每次推理都要多次检查 Context Window，累计延迟差异会很大。而且 Rust 的内存安全保证对于金融精度的成本计算至关重要——一个浮点误差就会导致金额算错。（加分点：我们还通过 PyO3 和 WASM 实现了跨语言复用，同一个 Rust 核心同时给 Python 后端和浏览器前端使用，避免了维护两套代码。）

**Q3: Context Window 满了怎么办？你的截断策略是什么？**

回答：使用分层截断。不是简单的"保留最后 N 条"，而是按优先级保留——System Prompt 必保、工具调用结果优先、最近对话次之、旧消息可弃。因为 System Prompt 定义了 AI 的行为边界，删掉它 AI 可能输出不安全内容；工具调用结果是 Agent 做决策的依据，删掉它 Agent 会出错。

**Q4: 你如何保证成本计算准确？**

回答：首先，用 Rust 的 Decimal 类型做金融精度计算，避免 IEEE 754 浮点误差。其次，成本计算考虑了四个因素：基础输入/输出价格、缓存命中折扣（省 90%）、长上下文溢价（GPT-5.5 >272K 翻倍）、思考 Token 成本（DeepSeek 等支持思考模式的模型有独立计费）。每个因素都用精确的模型定价表查表计算。

**Q5: 如果有人发 1GB 的文本过来，你的服务会挂吗？**

回答：不会。我们在所有输入路径都加了长度预检查——超过 1000 万字符直接拒绝并返回错误码。Rust 编译时保证没有缓冲区溢出，所以即使用恶意构造的输入也不会导致内存安全问题。这是用 Rust 而不是 C/C++ 的关键原因之一。

- 提到"定价表支持定期更新，不需要重启服务即可获取最新模型价格"
