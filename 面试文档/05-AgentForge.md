## 第一部分：是什么

### 1.1 生活类比：物流分拣中心

你在淘宝上买了三样东西：一本书（来自北京仓库）、一双鞋（来自广州仓库）、一个杯子（来自上海仓库）。

如果每买一样东西你都要分别联系三家仓库、三个快递员，你会疯掉。

所以有了**物流分拣中心**——你只需要下单，分拣中心自动：
- 识别每个商品应该从哪个仓库出货（= 适配器模式）
- 检查库存是否够、地址是否正确（= 中间件检查）
- 选择最便宜的快递方式（= 智能路由）
- 打包发货，记录每一步（= 轨迹追踪）

AgentForge 就是 AI Agent 的"物流分拣中心"。

### 1.2 大白话解释

**什么是"框架"？**

框架和库的区别：

| 概念 | 类比 | 谁控制流程 |
|------|------|----------|
| 库（Library） | 工具箱 | 你控制流程，你需要时拿工具用 |
| 框架（Framework） | 流水线 | 框架控制流程，你在指定环节填入你的东西 |

AgentForge 是框架——它定义了 Agent 执行的流水线（中间件管道），你只需要在各个中间件里填入你的业务逻辑。

**什么是"中间件管道"（Middleware Pipeline）？**

就像机场安检的流水线：

```
入口 → 检查护照 → 扫描行李 → 安检门 → 登机口
```

AgentForge 的中间件管道：

```
请求 → 1.Context检查 → 2.安全校验 → 3.缓存查询 → 4.限流 → 5.路由选择 → 6.记录追踪 → 7.成本累计 → 执行 → 响应
```

每一个中间件都可以：
- **前置处理**（请求进去前检查）——如"这个请求超过 Context 限制了吗？"
- **后置处理**（响应出来后加工）——如"这次调用花了多少钱？"
- **短路终止**（不符合条件直接拒绝）——如"检测到危险命令，拒绝执行"

**洋葱模型**：请求像剥洋葱一样一层层进去，响应再一层层出来。

```
请求 → [1→2→3→4→5→6→7] → 核心执行 → [7→6→5→4→3→2→1] → 响应
```

### 1.3 7 个中间件的职责

| # | 中间件 | 作用 | 依赖哪个项目 |
|---|--------|------|------------|
| 1 | ContextWindowCheck | 检查输入有没有超过模型记忆上限 | **token-core** |
| 2 | ToolCallSanitizer | 拦截危险命令（rm -rf 等） | — |
| 3 | SemanticCache | "这个问题之前问过吗？"缓存结果 | — |
| 4 | RateLimiter | 控制调用频率，防止超预算 | Redis |
| 5 | ModelRouter | 根据任务复杂度选最合适的模型 | **token-core** 定价表 |
| 6 | Tracer | 记录每一步的完整信息 | — |
| 7 | CostTracker | 实时累计成本 | **token-core** 计费 |

### 1.4 适配器模式：3 种框架，1 套接口

| 适配器 | 包装的框架 | 为什么需要 |
|--------|-----------|----------|
| LangChainAdapter | LangChain | 算法团队习惯用 |
| OpenAIAgentsAdapter | OpenAI Agents SDK | 工程团队习惯用 |
| NativeAdapter | 直接用 Anthropic/OpenAI SDK | 追求最低延迟时用 |

不管你用哪个框架，AgentForge 对外的接口都是一样的（`await agent.run(messages, tools, config)`）。换框架不需要改调用代码，只需改一个配置。

## 第二部分：为什么

### 2.1 为什么需要中间件管道而不是 if-else？

如果不用中间件管道，你需要这样写：

```python
if check_context(messages): return error
if is_dangerous(messages): return error
if rate_limited(): return error
# ... 7 个 if 嵌套
result = call_model(messages)
# ... 7 个后置检查
```

问题：
- **不可配置**——你想临时去掉"限流"中间件？改代码。
- **不可复用**——另一个项目也想用类似的管道？复制粘贴。
- **不可测试**——7 个逻辑混在一起，测一个要跑全部。

中间件管道的优势：
- **可配置**——YAML/JSON 文件决定管道有哪些中间件
- **可复用**——每个中间件是独立类，新项目可以挑着用
- **可测试**——每个中间件可以独立单元测试

### 2.2 为什么需要智能路由（ModelRouter）？

如果你把所有任务都交给最贵最好的模型（Fable 5，$10/百万输入），就像把所有快递都用顺丰次日达——大部分不紧急的快递浪费了钱。

| 任务类型 | 推荐模型 | 输入价格/百万 | 相对最贵的节省 |
|---------|---------|-------------|-------------|
| 简单分类/摘要 | Haiku 4.5 | $1 | 节省 90% |
| 中等推理/编码 | Sonnet 4.6 | $3 | 节省 70% |
| 复杂 Agent 任务 | Opus 4.8 | $5 | 节省 50% |
| 极限推理 | Fable 5 | $10 | — |

智能路由自动判断任务复杂度，选合适的模型。**就像打车软件根据距离自动选快车/专车/拼车。**

### 2.3 为什么需要 RL 训练桥接？

RL（强化学习）训练需要标准化的数据格式。Agent 执行的轨迹是自然语言形式，不能直接用于训练。

RL Bridge 把 Agent 轨迹转为 RL 训练样本：
- observation：Agent 当时看到了什么（Context Window 状态）
- action：Agent 做了什么（调用了什么工具）
- reward：做对了还是做错了（基于评测结果打分）

**类比**：把行车记录仪的录像（轨迹）转成驾校教练的评分表（训练数据）。

### 2.4 如何实现硬性要求

**高并发**：每个 Agent 在独立的 asyncio.Task 中运行，可以同时处理多个 Agent。Task 之间互不干扰，可以单独取消或超时终止。

**高可用**：主模型失败自动 Fallback——调用 Opus 失败 → 自动切换到 Sonnet → 再失败 → 切换到 Haiku。请求不会丢失。

**可扩展**：新增框架适配器只需实现 BaseAgent 接口（3 个方法）。新增中间件只需实现 BaseMiddleware（2 个方法）。不需要修改核心框架代码。

**低延迟**：gRPC（基于 HTTP/2 的二进制协议）用于内部高性能通信，比 REST API 快约 40%。token-core 通过 PyO3 直接调用（同进程，无网络延迟）。

**安全**：ToolCallSanitizer 检查所有输入是否包含危险命令。Prompt 注入防护——用户输入和 System Prompt 用 XML 标签严格分隔。

## 第三部分：怎么做

### 3.1 白话实现路径

```
1. 收到请求：POST /api/v1/agents/run
         ↓
2. 根据 framework 参数选择适配器（native / langchain / openai_agents）
         ↓
3. 构建中间件管道（7 个中间件按顺序排好）
         ↓
4. 请求进入管道：
    1. ContextWindowCheck: "输入有 50 万 Token，模型限制 100 万，Ok"
    3. SemanticCache: "缓存未命中，继续"
    4. RateLimiter: "这秒第 3 次调用，限制 10 次/秒，Ok"
    5. ModelRouter: "任务简单，选 Haiku 4.5 ($1/百万)"
         ↓
5. 核心执行：调用大模型 API
         ↓  
6. 响应出管道：
    7. CostTracker: "本次花费 $0.003"
    6. Tracer: 记录完整轨迹
         ↓
7. 返回 AgentResponse（含输出、步骤、成本、延迟）
```

### 3.2 智能路由如何判断任务复杂度

```
输入消息 → 提取特征：
  ├── 消息数量（>20 条 → 复杂任务）
  ├── System Prompt 长度（>2000 字 → 复杂约束）
  ├── 是否有工具调用（有 → 至少中等复杂度）
  └── 用户问题类型（简单问答 vs 多步推理）
         ↓
匹配模型级别：
  简单 → Haiku / GPT-4.1 Nano
  中等 → Sonnet / GPT-4.1
  复杂 → Opus / DeepSeek V4 Pro
  极限 → Fable 5 / GPT-5.5
```

## 第四部分：面试指南

### Q1: 中间件管道和普通函数调用链有什么区别？

普通函数调用链是硬编码的——A 调用 B，B 调用 C，顺序不能改。中间件管道是"声明式"的——你把中间件排成一个列表，框架帮你按顺序调用。关键区别：中间件可以动态增删、可以短路（不符合条件的请求直接返回，不再往下执行）、可以用洋葱模型前后处理。

### Q2: 为什么你的框架支持 LangChain 和 OpenAI Agents SDK？不是重复造轮子吗？

不是重复造轮子，是"统一接口"。LangChain 和 OpenAI Agents SDK 各有各的接口设计，直接混用会造成调用方式不一致。AgentForge 提供了一层抽象——适配器模式——让两个框架对外暴露相同的调用方式。这层的价值不在"功能"，而在"一致性"和"可替换性"。

### Q3: 智能路由如何避免选错模型？

选错有两种情况。一是选了太贵的模型处理简单任务——通过成本监控和历史分析发现。二是选了太差的模型处理复杂任务——执行失败后有 Fallback 机制自动升级到更强的模型。关键是：两个方向的错误都有自动修复机制，不需要人工干预。

### Q4: 你如何保证 Agent 执行的安全性？

四层防护：第一层——ToolCallSanitizer 拦截危险命令；第二层——RateLimiter 限制调用频率，防止失控；第三层——所有工具调用经 mcp-bridge 的 RBAC 和沙箱，隔离执行；第四层——Audit Log 记录所有操作，事后可追溯。

### Q5: 说说适配器模式和策略模式的区别？

适配器模式解决"接口不兼容"——把不同框架的接口转成统一的接口。策略模式解决"算法可替换"——同一个接口有不同实现。AgentForge 的适配器层是适配器模式（LangChain.run() → agentforge.run()），ModelRouter 是策略模式（cost_optimal / latency_optimal 是不同的路由策略）。
